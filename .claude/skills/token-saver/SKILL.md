# Token Saver — 作業トークン節約モード

このスキルが有効な間、以下のルールに従ってトークン消費を最小化して作業する。
節約のために正確性を犠牲にしてはならない。「読む量・出す量を減らす」のであって「確認を省く」のではない。

## 1. 読む前に検索する（Read の前に Grep/Glob）

- ファイル全体を読まない。まず Grep / Glob で該当箇所を特定し、必要な行範囲だけ `Read`（offset/limit 指定）で読む。
- Grep には `head_limit` を付け、`output_mode: files_with_matches` で足りるなら content モードを使わない。
- 探索目的で大きなファイル（数百行超）を丸ごと読むのは禁止。構造把握が必要なら、まず見出しやシンボル定義だけを検索する。

## 2. 同じものを二度読まない

- 一度読んだファイルを再読み込みしない。編集直後の確認再読も不要（Edit/Write が失敗すればエラーになる）。
- 会話内で既に確立した事実・調査結果を再取得しない。ユーザーが決めた方針を蒸し返さない。

## 3. コマンド出力を垂れ流さない

- 大量出力が予想されるコマンドはファイルへリダイレクトし、`tail` / `grep` で必要部分だけ見る。
  例: `npm test > /tmp/test.log 2>&1; tail -30 /tmp/test.log`
- ビルドログ・テストログ・JSONダンプを全文コンテキストに載せない。失敗行と要約だけ抽出する。

## 4. 広域探索はサブエージェントに委譲する

- 複数ファイル・複数ディレクトリにまたがる調査は Explore/general-purpose サブエージェントに任せ、結論だけ受け取る。親コンテキストにファイル内容を持ち込まない。
- 単発の事実確認（1ファイル・1シンボル）は自分で検索した方が安い。使い分ける。
- サブエージェントには「結論と該当パスだけ返せ。ファイル全文を貼るな」と指示する。

## 5. 出力を絞る

- 回答は要点先行・簡潔に。コードの全文引用や長い経過報告をしない。変更点は差分の説明だけで足りる。
- ツール呼び出しの合間の実況（「では次に〜します」）を書かない。発見・方針転換・ブロッカーのみ一文で報告。
- 完了報告は「何をしたか・結果・残課題」を数文で。ファイルごとの逐一の説明はしない。

## 6. 並列化と手戻り防止

- 依存関係のない読み取り・検索は1メッセージで並列実行する（往復回数＝コンテキスト再送回数を減らす）。
- 修正→検証は1サイクルで済むよう、修正前に失敗原因を特定してから直す。当てずっぽうの修正リトライはトークンの無駄。

## 7. 長時間作業はコンテキスト圧縮に耐える形で

- 長い作業では進捗・決定事項・残タスクを作業ファイル（例: `PROGRESS.md` やプランファイル）に逃がす。コンテキストが圧縮されても作業を再開できる。
- 中間成果物はチャットに貼らずファイルに書き、パスだけ報告する。

## 禁止事項（よくある無駄）

- ディレクトリ全体の `cat` / 再帰 `ls -R` の全文表示
- 「念のため」の関連ファイル一括読み込み
- 同一内容の要約を複数回書く
- 変更していないコードの長文引用
